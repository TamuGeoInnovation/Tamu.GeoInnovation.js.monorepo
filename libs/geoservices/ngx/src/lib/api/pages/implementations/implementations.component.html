Getting Started

The Geoservices Geocoding API is available to developers in several ways:

  • Raw HTTP, REST, or SOAP requests
  • Client JavaScript Libraries
    • CDN
    • NPM

Raw requests offer the greatest flexibility for developers allowing custom implementations based on their programming language of choice and consuming application formats (json, xml, csv, tsv, etc.)

Using the hosted version, through a CDN (Content Delivery Network), is best for applications that do not utilize a package manager like NPM. Simply add the script requirement to the head of your webpage and you're off to the races!

For projects utilizing the NPM package manager, module installation is done through its CLI and imported into your project. This has the benefit of including the API client library into your project and releasing network external dependencies (CDN) that could impact your application load times and performance. Furthermore, the client API libraries are written with TypeScript so developers get the added benefit of JSDoc or native TypeScript typings in supported IDE's or Code Editors.

The base API for all Geoservices API's is:

GET
https://geoservices.tamu.edu/Services/

Authentication

All Geoservices API's require an API key to process requests. If you have a Geoservices account, your key will be located in your account settings page. If you don't yet have an account, creating one is easy and free.

Please note that your API key is the secret key to your account and credit balance. Releasing your API key could lead to malicious parties using up your quota. Please ensure you don't expose your API key in public-facing web applications or store them in version control. Should your key become compromised, please contact us and we can issue a new API key.

Quota and Rate Limits

All registered users receive a daily quota of free credits that can be used for development and testing purposes. In order to ensure a consistent level of service quality for all of our users, all requests are rate-limited based on their subscription level.

Raw Queries

Our services can be called using any implementation that allows querying HTTP, REST, or SOAP resource endpoints. One of the simplest ways to make use of the API's through a front-end application is by building the request URL and making a call with the native Fetch API:

Browser support for the Fetch API is "good enough" for most applications, however if even broader support is required then querying the API with more lower-level XMLHttpRequests API is in order. Furthermore, both the Fetch API and the XMLHttpRequest API's are only available in browser environments and so they are not available to NodeJS developers working on non-browser projects. Node developers will have to use a third-party library for XHR requests.

Libraries & SDK's

Alternatively, we provide a small first-party, open-source, JavaScript client library (with support for for other languages in the future) for use in browser and Node applications to reduce the boilerplate required to create API requests and handle errors, facilitating service consumption. It provides XHR wrappers for the following services:

  • Geocoding
  • Reverse Geocoding
  • Address Parsing/Processing
  • Census Intersection
  • K-Nearest
